The Problem

ทำไมต้องสร้างใหม่?

"เราเคยเปิด Kahoot ในห้องเรียน — แต่ต้องสมัครสมาชิก, จ่ายเงิน, และข้อมูลทั้งหมดอยู่กับบริษัทต่างประเทศ"

❌ Kahoot ที่มีอยู่แล้ว

  • ต้องสมัครสมาชิกก่อนใช้
  • ฟีเจอร์ครบ = ต้องจ่ายเงิน
  • ข้อมูล Player อยู่กับ server ต่างประเทศ
  • ไม่สามารถติดตั้งบนเครื่องตัวเองได้

✅ VibeCheck

  • ไม่ต้องมี account
  • โอเพนซอร์ส — code เปิดสาธารณะ ดูและแก้ไขได้
  • ติดตั้งบน server ตัวเองได้ — ข้อมูลไม่ออกนอกองค์กร
  • เริ่มใช้งานได้ใน 30 วินาที
Host แชร์รหัส PIN 6 หลัก → Player เข้าจากมือถือ → เล่นได้เลย ไม่ต้อง login
How It Works

ระบบทำงานอย่างไร?

Host (เบราว์เซอร์) ──→ ระบบส่งข้อมูลสด (SignalR) ──→ จัดการเกม (GameService) ──→ Player (มือถือ)

ขั้นตอนเกม: รอ Player → โชว์คำถาม → รับคำตอบ → แสดงคะแนน → จบเกม

Tech Stack

  • .NET 10 Blazor Server — เว็บแอปฝั่งเซิร์ฟเวอร์
  • SignalR — ส่งข้อมูลแบบสด (real-time)
  • Tailwind CSS — ตกแต่งหน้าเว็บ
  • Web Audio API — เสียงเอฟเฟกต์
  • Docker → Render.com

ทำไมเลือก Blazor?

  • เขียน C# อย่างเดียว — ไม่ต้องแยก frontend/backend
  • มี SignalR built-in
  • นาฬิกา ควบคุมที่เซิร์ฟเวอร์ — โกงไม่ได้
  • โครงสร้างเรียบง่าย เหมาะทำใน hackathon

AI Workflow · Part 1

วางแผนก่อนลงมือ Code

เขียน SPEC.md — เอกสารตกลงทุกข้อก่อนเขียน code บรรทัดแรก

คำถามที่ต้องตัดสินใจก่อน

  • Player เชื่อมต่อใหม่ — เล่นต่อ หรือขึ้นเป็นคนใหม่?
  • Host หลุดออก — เกมหยุด หรือเดินต่ออัตโนมัติ?
  • เข้าระหว่างเกม — อนุญาต หรือบล็อก?
  • นาฬิกา — ควบคุมที่เบราว์เซอร์ หรือที่เซิร์ฟเวอร์?

คำตอบที่ล็อกไว้ก่อน build

  • เชื่อมต่อใหม่ → ขึ้นเป็น Player ใหม่
  • Host หลุด → เกมเดินต่ออัตโนมัติ
  • เข้าระหว่างเกม → บล็อก
  • นาฬิกา → ควบคุมที่เซิร์ฟเวอร์
"ถ้าไม่ตัดสินใจก่อน — ตอน code เสร็จแล้ว Host หลุดออก เกมจะค้างโดยไม่มีใครรู้ว่าต้องทำอะไร"
AI Workflow · Part 2

เขียน Code + ตรวจสอบ

ฟีเจอร์ที่ยากที่สุด: นาฬิกาถอยหลังที่ควบคุมจากเซิร์ฟเวอร์

บอก AI สิ่งที่ต้องการ → AI เขียน code → เราตรวจสอบ → แก้ไข → บันทึก

AI เสนอมา

  • ใช้ timer ของ .NET ส่งสัญญาณทุก 1 วินาที
  • กระจายไปทุกหน้าจอพร้อมกัน (broadcast)
  • หยุดทันทีเมื่อทุก Player ตอบแล้ว

เราตรวจพบและแก้เพิ่ม

  • มีโอกาส timer ทำงานซ้อนกัน (race condition) — เพิ่มการล็อกป้องกัน
  • ย้ายการส่งสัญญาณออกนอก block ที่ล็อกอยู่
"AI เสนอโครงสร้างที่ถูกทิศ — แต่ปัญหา timer ซ้อนกันต้องเราตรวจเจอและแก้เอง"
AI Workflow · Part 3

แก้บัก (Debug)

บักที่พบ

Player ตอบคำถามแล้ว — คะแนนบนหน้าจอ Host ไม่อัปเดตทันที

สาเหตุที่แท้จริง

คำสั่งส่งข้อมูลสดถูกเรียกขณะระบบล็อกตัวเองอยู่ — ทำให้ติดขัด (deadlock) บางครั้ง

วิธีแก้

  • ย้ายคำสั่งส่งข้อมูลออกมาก่อนจะล็อก
  • ล็อกเฉพาะส่วนที่เปลี่ยนสถานะเกมเท่านั้น

บทเรียน

  • AI แก้ผิดในครั้งแรก
  • เราต้องชี้แนะให้หาสาเหตุที่แท้จริง
  • ใช้ /gsd-debug ช่วยคิดอย่างเป็นระบบ
Learnings

บทเรียนจากการทำงานกับ AI

AI ทำได้ดี

  • ออกแบบโครงสร้างเกมได้ครบถ้วน
  • จัดหน้าตา UI ได้รวดเร็ว
  • เขียน logic นำเข้า/ส่งออก CSV
  • เสนอ confetti เอง โดยไม่ได้สั่ง!
  • ออกแบบระบบนาฬิกาได้ถูกทิศทาง

ต้องแก้เอง

  • timer ทำงานซ้อนกัน (race condition)
  • ส่งข้อมูลผิดกลุ่ม (SignalR group scope)
  • นาฬิกาเดินผิดจังหวะ 1 tick
  • ช่องโหว่ด้านความปลอดภัย

🔒 Security Review — OWASP Top 10

  • ไม่ตรวจสอบค่าเวลา CSV → แก้: จำกัดให้เลือกได้เฉพาะ 5, 10, 15, 20, 30, 60 วินาที
  • แสดง error ดิบให้ผู้ใช้เห็น → แก้: เปลี่ยนเป็นข้อความกลางที่ไม่เปิดเผยระบบ
  • SQL injection — ไม่มีฐานข้อมูล ใช้หน่วยความจำล้วน
  • XSS — Blazor แปลง output อัตโนมัติทุกจุด
  • ข้อมูลลับฝังใน code — ไม่พบ
